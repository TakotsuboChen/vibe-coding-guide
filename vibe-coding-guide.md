## 1. 核心心智模型

在讲任何操作之前，先建立三个会贯穿全文的心智模型。理解这三点，比记住任何命令都重要。

### 1.1 上下文窗口是首要约束

Claude Code 的"记忆"叫**上下文窗口（context window）**，大约 200K tokens。这个窗口在每次新会话开始时是空的，随着对话进行被填满。一旦接近上限，Claude 会开始"忘事"、变慢、答非所问。

官方文档把这一点说得直白：**"上下文窗口是你需要管理的最重要资源"**，并且 **"LLM 性能随着上下文填满而退化"**。

这和你硬件开发时的"工作内存"很像——你的脑子同时能记住的东西有限，项目越大越容易顾此失彼。Claude 也一样。所以整个指南的核心，其实都是围绕**怎么管理这个窗口**展开的。

### 1.2 交付能跑的切片，而不是零 bug 的完美代码

软件从来不是"没有 bug"才发布的。所有你正在用的软件都有几千个已知 bug。发布标准是"对用户来说够好"，不是"无 bug"。

**切片（slice）** = 一条从头到尾打通、用户能感知到价值的窄功能。

- 错误做法：先写完所有数据库层 → 再写完所有后端 → 再写前端。这是分层。每层写完之前你没有任何能用的东西，中间任何一刻停下来项目就流产了。
- 正确做法：做一个能跑的版本：一个输入框、一个按钮、点一下能存数据、能显示出来。丑、简陋，但从头到尾通了。

硬件类比：你不会因为 PCB 上还有可优化的走线就永远不投板。投板标准是"满足规格书"。Vibe coding 也一样——**目标是能用的切片，不是完美的代码**。

### 1.3 主动维护 > 自动魔法

你之前失败的一个核心根因，是期望 Claude Code 像数据库一样自动同步状态。真相是：

- CLAUDE.md **不会**自动更新，必须你主动说"更新它"
- `/clear` 后 Claude 会忘掉对话，只记得磁盘上的 CLAUDE.md 和 HANDOFF.md
- Claude Code **不会**帮你自动找 Skills 和 MCP

**这套体系的运转，靠的是你主动维护的几个文件 + 几个机械习惯，而不是任何"自动魔法"。** 好消息是：这些习惯不依赖代码能力，纯是纪律，业余开发者完全可以做到。

## 2. Claude Code 的文件与记忆体系

理解文件结构是理解一切的基础。你之前最大的困惑就在这里。

### 2.1 两层 `.claude` 目录

Claude Code 有两个 `.claude` 目录，作用完全不同：

| 层级 | 位置 | 作用 | 谁来写 |
|------|------|------|--------|
| **全局用户** | `~/.claude/` | 你个人所有项目通用的配置、记忆、skills | 你 |
| **项目级** | `项目/.claude/` | 这个项目专用的配置、skills、权限 | 你 |

全局 `~/.claude/` 下的关键内容：

```
~/.claude/
├── CLAUDE.md              ← 你个人的全局偏好（所有项目通用）
├── settings.json          ← 全局设置（权限、hooks 等）
├── skills/                ← 全局 skills（所有项目可用）
├── projects/              ← 各项目的会话记录 + 自动记忆
└── ...
```

项目级 `.claude/` 下的关键内容：

```
项目/
├── CLAUDE.md              ← 项目约定（团队/个人共享，进 git）
├── HANDOFF.md             ← 跨会话交接文档（进 git）
├── .claude/
│   ├── settings.local.json  ← 个人权限配置（gitignore）
│   └── skills/              ← 项目专用 skills
└── ...
```

### 2.2 那三个奇怪文件夹是什么

你看到 `~/.claude/projects/` 下有 `-home`、`-home-takotsubo`、`-home-takotsubo--claude` 三个文件夹。这是 **Claude 把你的工作目录路径"转码"后生成的**：把所有非字母数字字符（包括 `/`）替换成 `-`。

- `-home` ← 工作目录是 `/home`
- `-home-takotsubo` ← 工作目录是 `/home/takotsubo`
- `-home-takotsubo--claude` ← 工作目录是 `/home/takotsubo/.claude`

每个文件夹 = 一个"项目"的会话记录（`.jsonl`）+ 自动记忆（`memory/`）。

**这暴露了你之前的一个坏习惯**：你在 `~`（即 `/home/takotsubo`）直接启动 claude 做各种事，导致所有杂事和记忆都堆在 `-home-takotsubo` 下，混在一起。

### 2.3 正确的目录习惯

**特定项目**：放在 `~/projects/项目名/`，`cd` 进去再 `claude`。这样：

- 这个项目有自己的 `CLAUDE.md`、`.claude/settings.local.json`
- Claude 会给它生成独立的 `~/.claude/projects/-home-takotsubo-projects-项目名/` 记忆空间
- 和其他项目的记忆互不污染

**非特定对话**（比如查资料、问概念）：可以在 `~` 启动，但**不要在这里写真实项目代码**。

### 2.4 四层记忆配置

Claude Code 的配置/记忆有四层 scope（从宽到窄）：

| Scope | 位置 | 用途 | 进 git? |
|-------|------|------|---------|
| 用户全局 | `~/.claude/CLAUDE.md` | 个人跨项目偏好 | 不适用 |
| 项目共享 | `项目/CLAUDE.md` | 项目约定，团队共享 | 是 |
| 项目个人 | `项目/CLAUDE.local.md` | 个人对这个项目的笔记 | 否（gitignore） |
| 自动记忆 | `~/.claude/projects/<项目>/memory/MEMORY.md` | Claude 自己发现的笔记 | 不适用 |

**自动记忆**是 Claude 自己写的（你不用管），在会话开始时加载前 200 行或 25KB。其余三层是你维护的。

### 2.5 推荐的项目目录布局

```
~/projects/我的项目/
├── CLAUDE.md              ← 项目约定（你维护，进 git）
├── CLAUDE.local.md        ← 个人笔记（gitignore）
├── HANDOFF.md             ← 跨会话交接（你维护，进 git）
├── TODO.md                ← 以后想做的（防 scope creep）
├── README.md              ← 一句话说项目干嘛
├── .claude/
│   ├── settings.local.json  ← 个人权限（gitignore）
│   └── skills/              ← 项目专用 skills（可选）
├── src/                     ← 源代码
├── tests/                   ← 测试（项目变大后再加）
└── ...
```

`★ Insight ─────────────────────────────────────`
你之前困惑"是在 `~` 开始还是去项目目录"——答案是：**真实项目永远 `cd` 到项目目录再启动 claude**。Claude Code 没有"打开项目"的显式动作，它靠你 shell 的当前位置判断项目根。这就是为什么你之前在 `~` 开始会混乱——Claude 把 `~` 当成项目，所有杂事都堆在一个记忆空间里。
`─────────────────────────────────────────────────`

## 3. 项目启动：第一个 30 分钟

这是防流产最关键的一段。养成每次新项目都走一遍的习惯，项目就有了"生命"，不会中途莫名死亡。

### 3.1 30 分钟例行

```
1. mkdir 项目名 && cd 项目名 && git init
2. 写 README.md：一句话说这东西要干嘛
3. 让 Claude 写 CLAUDE.md：技术栈 + 第一个切片目标 + 编码约定
4. 写 TODO.md：把脑子里"以后要做"全倒进去，清空脑子
5. 让 Claude 实现最小切片（最蠢能跑版本，丑没关系）
6. /run 或 /verify 亲眼看它能工作
7. git add . && git commit -m "最小切片能跑"
8. 让 Claude 更新 CLAUDE.md + 写 HANDOFF.md：下一步是什么
```

### 3.2 为什么这 30 分钟能防流产

流产的本质是三件事：**做了一半没有能用的东西 → 下次打开忘了在干嘛 → 不想碰了**。

这套例行直接打这三点：
- 步骤 5-6：任何时候都有能跑的东西
- 步骤 3、8：任何时候停下来都能续上
- 步骤 4：回来时不用回忆，看 TODO 就行

30 分钟后你拥有：一个能跑的东西、一个存档点、一份交接说明、一份明确清单。项目已经不会流产了。

### 3.3 关键纪律：切片要小

**第一个切片一定要小到让你觉得"这太简单了吧"。**

- 想做一个博客：第一个切片 = 一个页面，能显示"Hello"，能存一篇文章标题到文件。**不**做用户系统、不做评论、不做富文本编辑器。
- 想做一个 CLI 工具：第一个切片 = 一个命令，跑起来能打印"hello"。**不**做参数解析、不做配置文件、不做帮助文本。

切片小的价值：它能在一次会话内完成、验证、commit。这保证你的上下文窗口不会爆，也保证你随时有一个"能用的版本"。

`★ Insight ─────────────────────────────────────`
新手最大的本能冲动是"一次做完"。这个冲动是流产的头号杀手。切片式开发反直觉——它要求你**故意只做一小部分，然后停下来**。但这是软件工程几十年验证下来的反脆弱结构：任何时候项目停住，它都是"能跑的半成品"而不是"没跑完的废墟"。硬件类比：先做出能上电点灯的最小系统，再加外设。
`─────────────────────────────────────────────────`

## 4. CLAUDE.md 与 HANDOFF.md 的正确用法

这是你之前失败的核心痛点。研究结论很明确。

### 4.1 CLAUDE.md 的真相

CLAUDE.md **不是**自动生成的数据库，也**不会**在会话中自动更新。它的本质是：

- **你写给 Claude 看的项目说明**，每次会话开始 Claude 都会完整读一遍
- `/clear` 之后，它是 Claude 知道"这个项目在干嘛"的唯一入口
- 内容应该是**相对稳定的项目级约定**，不是日志

关键事实：

| 问题 | 答案 |
|------|------|
| Claude 会自动生成 CLAUDE.md 吗？ | 不会。要么你手动写，要么 `/init` 生成草稿，要么直接对 Claude说"帮我写 CLAUDE.md" |
| 会话中 Claude 会自动更新它吗？ | **不会**。必须你主动说"把这条加到 CLAUDE.md"，或你自己编辑 |
| `/clear` 后会失忆吗？ | 会忘对话，但会重新读 CLAUDE.md。所以它是接续点 |
| `/init` 什么时候跑？ | 项目有一些代码之后跑才有意义（它靠分析代码库生成） |
| 空文件夹时怎么办？ | 不要 `/init`，直接对 Claude 说"帮我写 CLAUDE.md" |

### 4.2 CLAUDE.md 应该写什么

目标长度：**200 行以内**。它是稳定约定，不是日志。

推荐结构：

```markdown
# 项目概要
- 一句话：这是什么
- 技术栈：语言、框架、关键依赖
- 项目结构：src/ 还是 flat？测试放哪？

# 构建/测试/运行命令
- 安装依赖: npm install
- 运行: npm run dev
- 测试: npm test
- 构建: npm run build

# 编码约定
- 命名风格（camelCase / snake_case）
- 错误处理模式
- 日志规范

# 当前进度（这一节会更新）
- 已完成：登录切片（2026-07-10）
- 当前切片：文章 CRUD
- 下一步：评论功能
```

### 4.3 HANDOFF.md：社区事实标准

**这是研究里最重要的发现之一。** GitHub 上有 10+ 个仓库用 `HANDOFF.md` 这个约定（`STATE.md`/`progress.md` 零命中），这是社区验证的跨会话交接标准。

为什么需要 HANDOFF.md 单独存在，不能合并进 CLAUDE.md：

- **CLAUDE.md 是"稳定的项目约定"**（技术栈、编码风格，很少变）
- **HANDOFF.md 是"上次会话的临时状态"**（当前进度、下一步，每次更新）

你之前把它们混在一起，所以觉得更新 CLAUDE.md 像在写日志、不更新又失忆。分开就清爽了。

### 4.4 HANDOFF.md 模板

```markdown
# 项目交接

## 当前状态
- 已完成：[这个切片做到了什么]
- 当前切片：[正在做的一条窄功能]

## 下次会话第一件事
- [具体的第一步，比如"先跑 npm test 看是不是绿的"]

## 关键决策
- 选了 X 而不是 Y，因为...
- 放弃的方案：[避免下次重试死路]

## 已知坑
- [踩过的坑，带原因]

## 修改过的文件
- src/auth.ts, src/api/user.ts

## 验证命令
- npm test
- npm run dev
```

特别强调"放弃的方案"这一节——它防止下次会话重试死路，这是社区最看重的点。

### 4.5 维护节奏

**每次会话结束前**，花 2 分钟说一句：

> "更新 HANDOFF.md：今天做到哪、下次第一步做什么、踩了什么坑。同时更新 CLAUDE.md 的'当前进度'一节。"

这就是你的"检查点"。**养成这个习惯，比依赖 /compact 强一万倍。** 这是你之前失败最直接的补救——你以为它会自动更新，所以从没主动维护过它。

`★ Insight ─────────────────────────────────────`
CLAUDE.md vs HANDOFF.md 的分工，本质是"稳定 vs 易变"的分离。软件工程里这叫"策略与机制分离"——机制（技术栈、命令）很少变，策略（当前做哪块）天天变。把两者写进同一个文件，要么机制被频繁改坏，要么策略从不更新。分开成两个文件，各司其职。这个模式在工程里无处不在——Makefile 和 TODO.txt、电路图和BOM表、固件和配置。
`─────────────────────────────────────────────────`

## 5. 会话管理：/clear 与 /compact 的取舍

你之前大量用 /compact 并且觉得它非常不好用——你的体感完全正确，研究证实了为什么。

### 5.1 /compact 到底做了什么

`/compact` 把当前对话历史**替换成一段结构化摘要**。它不是"压缩无损保存"，而是**有损摘要**——信息会丢失。

### 5.2 为什么多次 /compact 会"慢性失忆"

每次 /compact 都会丢失信息，机制上有三层损失：

1. **摘要的摘要的摘要**：多次 /compact 时，第二次摘要的是"第一次摘要后的内容"，信息层层稀释
2. **skill 描述不会重新注入**：只有你实际调用过的 skill 被保留，其他 skill 的描述在 compact 后丢失
3. **嵌套目录的 CLAUDE.md 不自动重新注入**：项目根的 CLAUDE.md 会重新读，但子目录的不会

这就是你体感到"头部信息越来越被稀释"的根因。

### 5.3 正确的取舍：HANDOFF.md + /clear

Anthropic 官方把三大失败模式都列在文档里，**修复方式都是 /clear 而非 /compact**：

1. **Correcting over and over**（反复纠正）
2. **Kitchen sink session**（一个会话混多任务）
3. **Infinite exploration**（无限探索代码）

官方原文逐字说：

> A clean session with a better prompt almost always outperforms a long session with accumulated corrections.

（一个干净的会话 + 更好的初始提示，几乎总是胜过一个堆满纠正的长会话。）

### 5.4 对比

| | /compact | /clear + HANDOFF.md |
|--|----------|---------------------|
| 上下文状态 | 有损摘要，稀释 | 完全干净 |
| 风险 | 信息损失累积 | 无损失——你控制要带什么 |
| 适合 | 单任务中段喘息（最多一次） | 跨任务、跨会话边界 |
| 你的努力 | 自动 | 手动维护 HANDOFF.md |
| 质量 | 随次数退化 | 一致，因为你策展 |

### 5.5 正确的会话循环

```
┌─ 新会话开始 → Claude 自动读 CLAUDE.md + HANDOFF.md
│
├─ 做一个切片（一条能跑的窄功能）
│
├─ 切片完成 → /run 或 /verify 亲眼看它工作
│
├─ git add . && git commit -m "切片说明"
│
├─ 主动说："更新 HANDOFF.md 和 CLAUDE.md 的当前进度"
│
└─ /clear 开始下一个切片
```

### 5.6 什么时候 /clear

用 `/context` 命令看上下文占用（显示为彩色网格）。在这些时机 /clear：

- **一个切片完成并 commit 后**（最推荐）：干净开始下一切片
- **上下文到 200-300K 的"甜点区"之前**：不要等爆了再清
- **Claude 开始忘事、变慢**：立即清
- **想换方向**（从功能 A 转去功能 B）：先 commit + 更新 HANDOFF，再清

### 5.7 单次会话内 /compact 的合法用法

如果你必须在一个会话内继续，且上下文快满了：

- **最多 /compact 一次**
- 用带说明的 compact：`/compact 保留修改过的文件清单和测试命令`
- compact 完立刻去更新 HANDOFF.md，为下次 /clear 做准备

**绝对不要**多次 /compact 续命——这正是你之前失败的路径。

`★ Insight ─────────────────────────────────────`
/compact 和 /clear 的取舍，本质是"谁来做摘要"的问题。/compact 让 Claude 自动摘要，省事但失控；/clear + HANDOFF.md 让你手动摘要，费力但可控。业余开发者的关键优势是"人知道什么重要"——你比 Claude 更清楚哪些是死路、哪些决策关键。把摘要权拿回自己手里，是长期项目能延续的根本。
`─────────────────────────────────────────────────`

## 6. 最小够用的 Git 工作流

你是业余初学者，不用学全套 Git。这里只给你**vibe coding 需要的最小集**。

### 6.1 你只需要 8 个命令

按使用频率排序，每个一句话解释：

| 命令 | 作用 | 什么时候用 |
|------|------|-----------|
| `git init` | 在当前目录创建 `.git/`，开始追踪历史 | 项目第一次开始时跑一次 |
| `git status` | 看现在有哪些文件改了/新增了/还没追踪 | 每次 commit 前看一眼 |
| `git add .` | 把改动放进"暂存区"（待提交清单） | 改完一段、确认要存档前 |
| `git commit -m "说明"` | 把暂存区的改动永久存成快照 | 存档动作本身。说明写人话 |
| `git log --oneline` | 看历史提交列表，每条一行 | 确认刚才存进去了吗/想回到哪 |
| `git diff` | 看当前还没 add 的改动 | 确认 Claude 刚改了什么再决定要不要存 |
| `git restore <文件>` | 丢弃这个文件里还没 commit 的改动 | Claude 改崩了想撤销时——你的"读档" |
| `git switch -c <分支名>` | 创建并切到新分支；`git switch <名>` 切回 | 想试危险想法不影响主线时 |

> 老教程里你会看到 `git checkout -- <文件>` 和 `git checkout -b <分支>`，它们等价于 `git restore` 和 `git switch -c`。Git 2.23 之后官方推荐用 `restore`/`switch`，因为 `checkout` 一个命令干两件事容易出事。

### 6.2 不用 push / remote 行不行

**行，纯本地个人项目完全不需要 push。**

`git push` 是把本地 commit 发送到远程仓库（GitHub）。如果你只在本地一台机器上搞，从不协作，`init/add/commit/log/restore` 这一套就够用很久。

**唯一建议**：给本地仓库加个远程备份（另一台机器或 U 盘上的 bare 仓库），因为硬盘坏了 `.git` 也没了。但这是"备份"需求，不是工作流必需。

### 6.3 Commit 节奏——具体什么时候 commit

新手最容易抓不准。给你几条**机械规则**，照做就行：

1. **每个能跑起来的小功能完成时 commit**。不是"代码写完"，而是"我刚启动它，登录页能点了"——能验证它工作的那一刻就 commit
2. **让 Claude 做有风险的事之前 commit**。比如"重写整个数据库层""换框架"之前，先把当前能跑的状态存档
3. **`/clear` 之前 commit 一次**。清掉上下文后 Claude 不知道刚才干了什么，清之前 commit 等于给会话画句号
4. **每 30-60 分钟至少 commit 一次**，哪怕还在干活，让 Claude 先 commit 一个"work in progress"
5. **别等"完美"才 commit**。"代码还不够干净"不是不 commit 的理由。commit 是存档点，不是颁奖

**一句话总结：commit 不是"完成宣言"，是"存档点"。脏的存档点 >> 没有存档点。**

### 6.4 Git + /rewind 的双层安全网

Claude Code 里 `/rewind` 是更细粒度的"读档"——它不是撤销 git，是撤销 Claude 这一轮的改动，可以只回退对话不回退代码，也可以两者都回。

配合使用：
- **大改动前** `git commit`（项目级存档）
- **Claude 改崩了某一步** 用 `/rewind`（单步回退）
- **整个方向错了** `git restore`（丢弃未提交改动）或 `git switch` 切回旧分支

这三层网让你随时能退，是 vibe coding 不怕搞砸的底气。

`★ Insight ─────────────────────────────────────`
Git 的本质不是"版本控制"，是**给你的开发过程装上存档键**。对业余开发者，它的价值不在协作，而在"敢于放手让 Claude 试"——任何改动都可以 restore 回来。这就像硬件调试时有热风台随时能拆焊，你才敢上手试。没有 Git 的 vibe coding 等于没有 undo 键的编辑器，一次搞砸就劝退。
`─────────────────────────────────────────────────`

## 7. 最小够用的斜杠命令集

Claude Code 有 50+ 个斜杠命令，但业余开发者只需要这 10 个。其余的等你有具体需求再学。

### 7.1 你需要的 10 个命令

| 命令 | 干什么 | 你什么时候用 |
|------|--------|-------------|
| `/init` | 分析代码库生成 CLAUDE.md | 项目跑起来后跑一次 |
| `/context` | 看上下文占用多少（彩色网格） | 觉得 Claude 变笨时先看这个 |
| `/clear` | 清空对话重开，旧会话保留可 `/resume` | 切片完成后、换方向时 |
| `/rewind` | 回退到之前的代码/对话点 | Claude 改崩了，"读档" |
| `/diff` | 看当前改了什么 | commit 前看一眼 |
| `/run` | 启动 app 看它在跑 | 想眼见为实时 |
| `/verify` | 确认改动真的有效（端到端） | 准备 commit 前跑一次 |
| `/code-review` | 让子代理审当前 diff | 准备"出货"前 |
| `/resume` | 回到之前的会话 | 接着上次没做完的做 |
| `/branch` | 复制当前对话做实验 | 想试危险想法又不影响主线 |

### 7.2 暂时忽略的命令

这些等有具体需求再学，不要为了"用全功能"而硬学：

`/compact`（用 HANDOFF.md 替代）、`/fork`、`/background`、`/loop`、`/mcp`、`/plugin`、`/agents`、`/hooks`、`/permissions`、`/add-dir`、`/cd`、`/sandbox`、`/plan`、`/goal`、`/skills`。

> `/plan`、`/goal`、`/skills` 等其实是很有用的命令，但对你现阶段不是必需。等你跑通几次完整流程、手感建立后再探索。

### 7.3 为什么不多学

**业余开发者 80% 的需求只要这 10 个命令。** 其余命令要么是给专业团队协作（`/mcp`、`/agents`、`/permissions`），要么是高阶自动化（`/loop`、`/fork`、`/background`）。

学命令有认知成本——每多学一个，你脑子里就多一件"我是不是该用 X"的判断负担，反而拖慢主线。先用熟这 10 个，等某个具体痛点出现再按需学。

`★ Insight ─────────────────────────────────────`
工具掌握有个反直觉规律：**用熟少量工具的人，比浅尝大量工具的人产出高**。这不是因为"少即是多"的玄学，是因为每件工具都有"激活成本"——你要记得它、判断该不该用、用错了还要回退。10 个用熟的命令 + 1 个清晰的 CLAUDE.md，比 50 个半生不熟的命令 + 空白 CLAUDE.md 强得多。这就像你焊台再多，不如把一台风台和烙铁用熟。

## 8. Skills 与 MCP：少即是多

GitHub、B 站每天推成百上千的 Skills，你不懂代码就不知道该用哪个——这个困惑本身是对的。**破解点不是"学更多"，是"装更少"**。

### 8.1 装多了 Claude 反而变笨——有量化证据

这不是玄学，有具体证据：

- 一个 MCP 工具的 schema 可能吃 **50K+ tokens**（在你说第一句话之前就没了）
- 有人发现装一堆 MCP 后 **16K tokens 在打字前就消耗掉了**
- HN 上有个项目把 315KB 的 MCP 输出压缩到 5.4KB，会话时间从 30 分钟延到 3 小时
- Skills 多了，描述会被截断、关键词丢失，Claude 会**选错工具**

你体感到的"Claude Code 裸奔形式直接开始做、变傻"，根因之一就是装太多 Skills/MCP 把上下文撑爆了。

### 8.2 Skills 的层级

| 位置 | 路径 | 作用域 |
|------|------|--------|
| 个人全局 | `~/.claude/skills/<name>/SKILL.md` | 你所有项目 |
| 项目级 | `项目/.claude/skills/<name>/SKILL.md` | 这个项目 |
| 插件 | 插件自带的 skills | 插件启用范围 |

Claude 在会话开始时加载所有 skill 的**一行描述**（约 450 tokens），实际全文只在被调用时才加载。Skills 多了，描述列表被截断，关键词丢失，misfire 风险上升。

### 8.3 MCP 的层级

| Scope | 配置位置 | 共享? |
|-------|---------|-------|
| 项目 | `项目/.mcp.json` | 是（进 git） |
| 用户 | `~/.claude.json` 的 `mcpServers` | 否 |
| 本地 | `~/.claude.json` 的 `projects/<path>/mcpServers` | 否，单项目私有 |

### 8.4 你的 Starter Pack（3 个以内）

**Skills：**
- `dataviz`（你已经有了，画图用）
- 官方自带的 `/init`、`/verify`、`/run`、`/code-review` 已经够用
- **暂时不要装更多**

**MCP（最多 2 个）：**
- `context7`（拉最新库文档，对你这种不熟 API 的人友好）
- 至多一个领域相关的（比如硬件场景的 datasheet MCP）

### 8.5 Superpowers 要不要装？

**研究结论：暂时不要。**

理由：
- 它是"完整软件开发方法论"（brainstorm→plan→TDD→subagent→review 全流程），假设用户能评估 plan、做 spec 合规 review
- 官方在 Best practices 里有平衡意见："一句话能描述的 diff 就跳过 plan"——和 Superpowers 的强制流程相反
- 对纯非编码者，每个小修复都走全流程会**让小任务变重**，反而增加中断和复杂度
- 等你积累出"我看得懂这个项目结构"的感觉再考虑

### 8.6 Skills 更新怎么办？

- 官方机制：改 SKILL.md 文件**当会话内热生效**，不用重启
- 代码不会因为 skill 更新变屎山——skill 只是指导 Claude 怎么做，**代码本身在 git 里**
- 担心 skill 指导不好？用 git commit 隔离：每个切片 commit，坏 skill 指导的代码就一个切片，`git restore` 回退即可

### 8.7 你现在自制的 skills 对不对？

存在 `~/.claude/skills/<name>/SKILL.md` 就是正确格式。可以在 `/skills` 里列出来看。自制 skills 是合法的，不用非得从网上下载。

### 8.8 破解死循环

你陷入的死循环：不懂代码 → 不会选 Skills/MCP → Claude 更傻 → 产出更差 → 依然不懂代码。

破解点**不是"学更多 Skills"**，而是"先做对基础"。裸奔的 Claude Code + 好的 CLAUDE.md + HANDOFF.md + git + 切片，比装满 Skills 的 Claude Code 强得多。

**Skills 是乘数，但你基础是 0 时，乘数再大也是 0。先把基础做对，再加 Skills。**

`★ Insight ─────────────────────────────────────`
每个 Skill/MCP 都有三重成本：上下文 token、认知负担、误触发风险。业余开发者最稀缺的是上下文窗口和注意力，不是工具数量。这和 PCB 设计同理——元件越多故障点越多，简洁的设计反而可靠。当别人炫耀装了 20 个 MCP 时，你要知道那是在给上下文窗口塞垃圾。克制就是竞争力。
`─────────────────────────────────────────────────`

## 9. Bug 螺旋逃生：最关键的一章

这是你之前所有项目流产的直接原因。这一章请反复读。

### 9.1 你的失败模式被官方文档直接命名了

Anthropic 官方 Best practices 把三个失败模式列在一起，**你的螺旋就是这三个的叠加态**：

1. **Correcting over and over**（反复纠正）
2. **Kitchen sink session**（一个会话混多任务）
3. **Infinite exploration**（无限探索代码）

官方原文逐字解释了为什么无限找 bug 是**数学上必然**的：

> A reviewer prompted to find gaps will usually report some, even when the work is sound, because that is what it was asked to do. Chasing every finding leads to over-engineering.

（一个被要求"找问题"的 reviewer 总会报告一些问题，即使代码没问题——因为这是它被要求做的。追每个发现会导致过度工程。）

### 9.2 为什么"再查一遍"永远有 bug

**核心机制**：每次"修 bug"本身就是一段新代码改动 = 一个新 diff = 新的可审查面。

你让它"审查所有代码"，它审的是刚改过的代码，改动又产生新的可挑毛病的地方。这是一个**不动点不存在的递归**——除非代码完全不变，否则总能找到"可以更严谨"的地方。

所以 N 轮之后你既不会得到"零 bug"，也不会得到"明显收敛"，你会得到**一个看起来永远有下一件事要做的列表**。这不是 Claude 笨，是任务定义本身错了。

硬件类比：你让 QA"再测一遍板子找问题"，每次修复后重新测，永远能找到可以优化走线的地方。投板标准是"满足规格书"，不是"走线完美"。

### 9.3 正确的心智模型

**Bug 实际上是无限的。目标不是"零 bug"，目标是"一个能工作的切片"。**

- 软件从来不是"没有 bug"才发布的。所有你用着的软件都有几千个已知 bug
- "找 bug"是**发散任务**（无穷可生成），"做出能用的东西"是**收敛任务**（有终点）
- 你用发散工具去追收敛目标，永远到不了
- **关键转换：把"找 bug"换成"跑有终止条件的检查"**——跑测试、跑 build、`/verify`。检查通过即停，不是"再 review 一轮"

### 9.4 逃生清单（机械规则，照做）

1. **两轮规则**：让 Claude 审查最多**两轮**。第二轮还在挑的问题，记进 TODO.md，**不在当前会话修**。继续修就进入"修→审→再修"的死循环

2. **严重度规则**：只修"会让当前切片不能用的 bug"。边缘情况、风格、性能、"以后可能出问题"——全部进 TODO，不在当下修

3. **新 bug 计数规则**：如果一轮审查找到的"真 bug"数量比上一轮还多，**立即停**。这通常意味着上一轮的"修复"在引入新问题，继续只会更糟

4. **换"找 bug"为"跑检查"**：给 Claude 一个返回 pass/fail 的检查（测试/build/截图/`/verify`），检查通过即停

5. **限制 reviewer 范围**：明确说——"只报影响正确性的问题，其他当可选"。这是官方原文的用法

6. **用 `/code-review` 而不是自然语言审查**：它在 fresh 子代理里审当前 diff，不受主会话历史污染，reviewer 不会因为"刚刚一起写过"而偏袒

7. **时间规则**：在一个切片上"审查+修"超过总时间的一半，就停。剩下时间应该用来推进下一个切片或 commit

### 9.5 为什么"扫所有代码找 bug"是反模式

因为它把你的注意力从"交付功能"转到"清查代码库"。对非程序员的个人项目，**代码库整洁度不是你的交付物，能用的功能才是**。

"扫一次全库找 bug"的正确用途只有一种：**接手别人/过去的自己写的陌生代码时做一次摸底**，摸完就停，别养成习惯。

### 9.6 你之前的具体失败路径

梳理一下你描述的失败路径：

```
让 Claude 按 Plan 执行 → 找到 Bug → 修复
  ↓
担心有别的严重 Bug → 让 Claude 审查所有代码（自然语言）
  ↓
找到 Bug → 修复 → 再说一遍"审查所有代码"
  ↓
又能找到 → 重复十几次还能找到
  ↓
同一个会话中重复多遍 → 上下文堆积 400-500K → 性能严重下降
  ↓
想 /compact 或 /clear → 遇到失忆问题（见第 5 章）
  ↓
Claude 提议"从头构建"或"修复部分先跑起来"
  ↓
你已经卡住、失去信心 → 觉得继续也是屎山 → 项目流产
```

这个路径里每一步都有对应的破解，分散在前面的章节里：

- "审查所有代码" → 第 9.4 节两轮规则 + /code-review
- "上下文 400-500K" → 第 5 章 /clear + HANDOFF.md
- "Claude 提议 rebuild vs patch" → 第 10 章决策框架
- "失忆" → 第 4 章主动维护 CLAUDE.md/HANDOFF.md

`★ Insight ─────────────────────────────────────`
Bug 螺旋的本质，是你把"收敛任务"（做出能用的切片）误用"发散工具"（无限找问题）去推进。这个陷阱不只在你身上——Anthropic 官方文档专门写 Callout 警告它，说明这是所有 Claude Code 用户最常见的失败模式。知道这一点后，你下次再想"再让 Claude 查一遍"时，会有一个声音提醒你：停下来，跑个检查就够，剩下的进 TODO。这个"停下"的肌肉记忆，是业余开发者最重要的 vibe coding 技能。
`─────────────────────────────────────────────────`

## 10. Rebuild vs Patch 决策框架

当 Claude 说"我可以从零重写（大工程）"或"打补丁继续"时，用这几条**不需要懂代码也能套**的启发式。

### 10.1 倾向 Patch（打补丁）的情况

- **能跑的代码不少**。已经有几百行能工作的代码 → 重写等于把战果扔掉。Joel Spolsky 的经典论点：旧代码里那些"看起来很丑的毛"往往是真机调试出来的边界修复，扔掉等于把这些坑重新踩一遍
- **bug 在边缘**，不在骨架。比如"某个按钮点两次会重复提交"——这是边角，patch 掉就行
- **架构是对的，只是实现脏**。数据流、模块划分大体合理，只是代码丑 → 局部重构，不要推倒
- **你是新手，重写对你风险更高**。重写需要你脑子里有清晰的目标架构，你还没有。打补丁是在已有结构上小修，认知负担低

### 10.2 倾向 Rebuild（重写）的情况

- **几乎没有能跑的代码**。只有几十行实验性代码、还没形成任何用户价值 → 重写成本极低，扔掉不可惜
- **架构本身就是错的**。比如选错了技术栈（该用数据库却用了文件、该是事件驱动却写成了轮询）。这种 bug 不在某一行代码，而在地基，patch 不动
- **同一个问题反复以不同形式出现**。修了这个那个又坏，说明根在架构。**判断法**：连续 3 次以上"修 A → B 坏 → 修 B → C 坏"的连环崩，就别 patch 了
- **上下文已 400-500K**：已经超过 200K 硬窗口意味着你已多次 compact，结构信息已丢失
- **你完全看不懂现在的代码在干嘛**（谨慎用这条：新手看不懂 ≠ 代码烂，可能是你还没学到。让 Claude 先解释一遍，再判断）

### 10.3 一句话决策法

> **"有多少能跑的代码" × "bug 在地基还是在边角"：能跑的多且 bug 在边角 → patch；能跑的少或 bug 在地基 → rebuild。**

新手默认倾向 **patch**。重写是高阶操作，需要更多判断力，在你积累出"我看得懂这个项目结构"的感觉之前，能 patch 就 patch。

### 10.4 Rebuild ≠ 丢代码

这是最关键的一点。**Rebuild 不等于把旧代码删了重写。** 用 Git 的分支：

```bash
git switch -c rebuild-v2   # 新分支重写
# 在新分支里重新生成代码...
# 万一重写更糟：
git switch main            # 切回旧分支，旧代码完好
```

旧分支留着，重写在新分支做，任何时候都能切回来。这消除了"重写 vs patch"的心理负担——你可以两个都试。

## 11. 测试与验证：非程序员的实用法

### 11.1 "测试"到底是什么

对新手来说，"测试"就是回答一个问题：**"改了这段代码之后，我怎么能知道它还能用？"**

分两层：

- **手动测试**：你亲自把程序跑起来，点一下、输个数据、看结果对不对。**新手一开始这样完全够**
- **自动化测试**：写一段"用来检查程序"的代码，以后每次改完自动跑一遍，几十秒告诉你有没有把旧功能弄坏

### 11.2 我需要正式写单元测试吗？

**前几个项目：不需要。** "跑起来、目测能工作"对个人玩具项目完全合格。

自动化测试的价值在于"重复手动检查变得不可行"之后才显现——项目小、你一个人、改动不多时，手动检查的成本比写测试低。

**什么信号说明该开始写测试了：**

- 你发现自己**反复手动点同一套流程**确认没坏 → 该自动化了
- 你**不敢改老代码**，怕一动就崩 → 该有测试兜底了
- 项目超过**几十个文件**，或者有**数据库/外部 API** 这类容易出状态错的东西
- 你开始**频繁发布**给别人用，回归 bug 成本变高

硬件类比：单元测试像 PCB 上的测试点。原型板你拿万用表到处戳就行；量产了你就要留好测试点、写自动测试程序。项目复杂度上来了，测试点才值得留。

### 11.3 /verify 和 /run 是干嘛的

- **`/run`**：启动并驱动你的 app，**让你看到一个改动真的在工作**——它操作真实运行的程序，不只是跑测试
- **`/verify`**：构建并运行你的 app，**确认一个代码改动确实做到了它该做的事**，且不 fallback 到"只跑测试或类型检查"。强调端到端跑一遍看行为

新手什么时候用：

- **`/run`**：你想"看一眼这东西现在长什么样/能不能跑"。改完 UI、改完逻辑后想眼见为实 → `/run`
- **`/verify`**：你或 Claude 刚改完一段非平凡代码，想确认它确实干了该干的事 → `/verify`
- **准备 commit 前**跑一次 `/verify` 是好习惯——它是"手动测试自动化版"
- **只改了文档/测试/纯无运行界面的代码时不要用** `/verify`（没有东西可观察）

`★ Insight ─────────────────────────────────────`
新手常常把"测试"想得太正式，觉得是要写测试框架、学 pytest/Jest 之类。其实测试的本质就是"用一个客观的、可重复的检查，替代你肉眼看一遍"。从"每次改完手动点一遍"开始，等痛点出现再升级到自动化。这个演进路径和硬件测试一样——先万用表，再测试夹具，再自动测试机。不要一上来就上自动测试机。
`─────────────────────────────────────────────────`

## 12. 长期项目防流产清单

### 12.1 提交节奏

- [ ] 每个 working slice 完成 → commit
- [ ] 每次让 Claude 做危险动作前 → commit
- [ ] `/clear` 前 → commit
- [ ] 每 30-60 分钟至少一次 commit，哪怕是 WIP
- [ ] commit message 写人话，别写 "fix" / "update"，写"加登录页，还没接后端"

### 12.2 切片纪律

- [ ] 项目里**永远有一个能跑的版本**，哪怕只能跑一个最蠢的功能
- [ ] 下一个 slice 只加**一条**新功能，不一次加五条
- [ ] 一个 slice 没跑通之前，不开始下一个
- [ ] 第一个 slice 要小到让你觉得"这太简单了吧"

### 12.3 交接文档

- [ ] 项目根目录放 `CLAUDE.md`（稳定约定）+ `HANDOFF.md`（当前状态）
- [ ] 每次收工时花 2 分钟说："更新 HANDOFF.md 和 CLAUDE.md 的当前进度"
- [ ] HANDOFF.md 必写"下次会话第一件事"和"放弃的方案"
- [ ] commit message 写人话

### 12.4 /clear 时机

- [ ] 一个 slice 完成并 commit 后 → /clear
- [ ] 对话明显变慢、Claude 开始忘事 → /clear
- [ ] 想换方向 → 先 commit + 更新 HANDOFF，再 /clear
- [ ] 用 `/context` 看占用，到 200-300K 甜点区前主动 /clear

### 12.5 范围纪律（防 scope creep）

- [ ] 把"以后想做但不属于当前 slice 的想法"写进 `TODO.md`，**不立刻做**。这一条是防流产最有效的
- [ ] 一个 slice 的定义写在一句话里，超出这句话的"顺便改一下"全部进 TODO
- [ ] 每次开新会话先问自己：今天要做的那一条 slice 是什么？只做它

### 12.6 Bug 审查纪律

- [ ] 审查最多两轮，第二轮的问题进 TODO
- [ ] 只修影响当前切片的 bug，其他进 TODO
- [ ] 用 `/code-review` 而不是自然语言审查
- [ ] 新 bug 比上轮多 → 立即停
- [ ] 把"找 bug"换成"跑检查"（测试/build/`/verify`），检查通过即停

### 12.7 为什么这套能防流产

流产的本质是"做了一半没有能用的东西，下次打开已经忘了在干嘛，不想碰了"。

切片 + commit + 交接文档 + 范围纪律四件套直接打这三点：
- 任何时候都有能用的东西（切片 + commit）
- 任何时候停下来都能续上（交接文档）
- 回来时不用回忆（TODO + HANDOFF）
- 不会越做越乱（范围纪律 + bug 审查纪律）

## 结语：你的下一步

### 你之前失败的根本原因

不是不懂代码。是缺三件套：

1. **CLAUDE.md/HANDOFF.md 主动维护**（你以为是自动的，其实不是）
2. **git commit 节奏**（你之前没有存档点）
3. **切片式推进**（你想一次做完，结果做一半就崩了）

这三样不依赖代码能力，纯是习惯。把它们做对，裸奔的 Claude Code 就够用；做不对，装再多 Skills/MCP 也救不了。

### 给你的建议路径

1. **先把这份指南读两遍**，特别是第 9 章 Bug 螺旋
2. **挑一个你真的想做的小项目**（命令行小工具、个人网页都行，不要选大的）
3. **严格按第 3 章的 30 分钟例行**启动它
4. **每个切片完成后**，严格按第 12 章清单过一遍
5. **跑完 3-5 个切片**，你会建立"我能让一个项目活着"的信心
6. **之后再考虑**装 Skills、加 MCP、学更多斜杠命令

### 一句话总纲

**CLAUDE.md 记录项目状态 + Git 做安全网 + HANDOFF.md + /clear 控制上下文 + 切片式推进 = 业余开发者的项目可以长期延续。**
